# PROJECT CARD ACTIONS

When hovering over a project object, reveal the project actions.

Example:

┌─────────────────────────────┐
│                             │
│          FORGEX             │
│                             │
│   Digital Forensics         │
│                             │
│   [ LIVE DEMO ↗ ]           │
│   [ VIEW SOURCE ◇ ]         │
│                             │
└─────────────────────────────┘

The buttons should be small floating clay controls.

The Live Demo button should communicate:

"Open the real deployed project."

The View Source button should communicate:

"Inspect the source code on GitHub."

Use an external-link icon for Live Demo.

Use a GitHub icon for View Source.

---

# SPATIAL PROJECT INTERACTION

When the user hovers over a project:

1. The project object rises toward the viewer.
2. The object rotates subtly toward the cursor.
3. The clay surface becomes more prominent.
4. The project preview expands.
5. The project description fades in.
6. The two action buttons emerge from the surface.

The buttons should feel like they are physically attached to the project object.

Do not make the buttons appear as ordinary flat HTML buttons.

---

# PROJECT CASE STUDY ACTIONS

When a project is opened into its detailed case study, place the two primary actions near the project title.

Example:

FORGEX

Digital Forensics / Security / Open Source

The Live Demo button opens:

https://landing749.github.io/ForgeX/

The View Source button opens:

https://github.com/Landing749/ForgeX

The GitHub link MUST open the exact repository for that project.

# PROJECT FOOTER ACTIONS

At the end of every project case study, repeat:

┌──────────────────────────────────────────────┐
│                                              │
│               EXPLORE THE PROJECT            │
│                                              │
│        [ OPEN LIVE DEMO ↗ ]                  │
│                                              │
│        [ VIEW SOURCE ON GITHUB ◇ ]           │
│                                              │
└──────────────────────────────────────────────┘

These should be large tactile claymorphic buttons.

The buttons should have strong depth and soft shadows.

On hover:

* Move toward the viewer
* Increase shadow
* Slightly scale
* Change surface highlight

On click:

* Give a subtle press-down effect
* Then open the external link

---

# GITHUB REPOSITORY VISIBILITY

The GitHub repository should be visible in the portfolio, not hidden.

The portfolio should communicate that these are not just visual mockups.

The visitor should be able to:

1. See the project.
2. Explore the live application.
3. Inspect the source code.

This is especially important for technical projects.

Use the following hierarchy:

**PROJECT**

↓
**SEE IT IN ACTION**

↓
**EXPLORE THE SOURCE**

The Live Demo and GitHub actions should be available from:

* Project hover state
* Project detail view
* Project case study
* Project footer

Do not force the visitor to search for the repository.

---

# GITHUB VISUAL TREATMENT

The GitHub button should use the GitHub icon.

Label:

**VIEW SOURCE**

or

**VIEW ON GITHUB**

Do not use "GitHub" as the only text if the meaning is unclear.

The button should visually communicate that clicking it opens the project's source code.

The overall experience should reinforce:

**I build real projects, deploy them, and make the source available.**
